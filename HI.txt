standardized keys.
Each key represents the presence, absence, or classification of mandatory
documents and clinical findings as per the STG reference.
Note:
• Solutions that do not strictly adhere to the specified output format will
be rejected without evaluation.
• Solution is only acceptable if it is solved for all 4 packages.

A.1. Case & Document Identification
Key: case_id
• Description: Unique identifier for the case
• Source: Folder name
• Type: String
Key: S3_link
• Description: S3 link of the document being analyzed
• Type: String (URL)
Key: procedure_code
• Description: Treatment / Package code
• Expected Value: "MG064A"
• Type: String
• Rule: Must match the assigned package exactly
Key: page_number
• Description: Page number within the document
• Type: Integer
• Rules:
o If the document is a single-page image, page number is always 1
o If the document is a multi-page PDF, page number increments
sequentially
A.2. Mandatory Clinical Documents (Presence Check)
STG Reference: Page 3 – PPD / CPD, Section 2.2
Use 1 = Present, 0 = Absent
Key: clinical_notes
• Reference: Page 3, PPD, Section 2.2.a
• Description: Clinical note documenting patient condition
• Mandatory: Yes
Key: cbc_hb_report
• Reference: Page 3, PPD, Section 2.2.b
• Description: CBC / Hemoglobin investigation report
• Mandatory: Yes
Key: indoor_case
• Reference: Page 3, CPD, Section 2.2.a
• Description: Indoor case / admission documentation
• Mandatory: Yes
Key: treatment_details
• Reference: Page 3, CPD, Section 2.2.b
• Description: Treatment or transfusion details
• Mandatory: Yes
Key: post_hb_report
• Reference: Page 3, CPD, Section 2.2.c
• Description: Post-treatment Hemoglobin report
• Mandatory: Yes
Key: discharge_summary
• Reference: Page 3, CPD, Section 2.2.d
• Description: Discharge summary after treatment
• Mandatory: Yes
A.3. Clinical Condition Classification
Key: severe_anemia
• Reference: Section 3.2 (STG Criteria)
• Type: Binary
• Rule:
o 1 → Severe anemia criteria satisfied
o 0 → Criteria not satisfied
A.4. Clinical Signs Assessment
STG Reference: Pages 1–2, Section 1.2 – Clinical Key Pointers
Mark 1 if present, 0 if absent
Key: common_signs
• Reference: Pages 1–2
• Examples: Pallor, fatigue, weakness (as per STG)
• Rule:
o If any common sign is documented, mark 1
o If no common signs are documented, mark 0
Key: significant_signs
• Reference: Page 2
• Examples: Tachycardia, breathlessness, etc.
• Rule: Same as above
Key: life_threatening_signs
• Reference: Page 2
• Examples: Cardiac failure, severe hypoxia, shock
• Rule: Same as above
A.5. Document Completeness Check
Key: extra_document
• Description: Presence of documents outside STG-listed requirements
• Type: Binary
• Rule:
o 0 → No extra documents
o 1 → Any additional document present (e.g., consent forms,
unrelated lab reports)
A.6. Document Timeline Ranking
Key: document_rank
• Description: Chronological position of the document
• Type: Integer
• Rule: Assign ranking based on clinical timeline
• Rank 99: Assign 99 Rank in case of Extra Documents
Note:
If a document spans multiple pages (e.g., a multi-page PDF such as an investigation report), the
same document_rank must be assigned to all pages of that document.
For example, if an investigation report is ranked as 3, then all pages belonging to that
investigation PDF must have document_rank = 3.
Typical Order Example
1. Clinical Notes
2. Investigations (CBC / Hb reports)
3. Treatment Details
4. Post Hb Report
5. Discharge Summary
• Lower number = earlier document
• Higher number = later document
• Rank 99=Extra Document

This is the detailed description of the ps1-> Problem Statement Title

Automatically read mixed quality healthcare documents, extract key data, detect mandatory visual elements (like stamps / signatures), and check compliance with standard treatment/policy rules — then produce an explainable Pass / Conditional / Fail decision with reasons based on Standard Treatment Guidelines (STG) provided by NHA for each package code.



Core Problem Statement

Reads documents even if blurry or in different languages.

Finds important fields (patient name, diagnosis, procedures, amounts).

Detects visual cues (hospital stamp, doctor signature, implant stickers, QR/barcodes).

Checks claims against rules (eligibility, diagnostics, length of stay, package logic).

Explains decisions with confidence and evidence source.

Constructs an episode timeline and validates chronology.

Highlights extra or non-required documents.



Expected Solution

Document intake: PDFs/images (low to high quality), multilingual.

OCR & layout understanding: text + structure (sections / tables / line items).

Visual element detection: stamps, signatures, implant stickers, QR/barcodes, invoice items.

Data structuring: standard schema (patient, diagnosis, procedures, dates, costs).

Rules engine: encode STG/policy checks (eligibility, contraindications, length of stay).

Explainability: confidence scores + provenance (page number, bounding box, source doc ID).

Decisioning: Pass / Conditional (needs more info) / Fail with prioritized flags.

Problem Statement 1





AI to process healthcare claims from PDFs/images, extract key data, verify STG compliance, detect visual proofs, flag extras, build timelines, and output explainable pass/conditional/fail decisions

DatasetHackathon

Building an AI-assisted claim validation system for healthcare documents that can read mixed-quality PDFs, scans, and images, extract important medical and administrative details, detect mandatory visual evidence like stamps, signatures, implant stickers, and QR/barcodes, and then check whether the submitted claim complies with package-specific Standard Treatment Guidelines (STG). The system must also generate an explainable Pass, Conditional, or Fail outcome, identify unnecessary extra documents, and construct a chronological treatment timeline from admission to discharge.
